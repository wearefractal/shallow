00185    spots and handles magically.  Don't look too closely at them as
00186    they will rot your brain.
00187 
00188    The assembly code sequences for all architectures is in this one
00189    file.  This is because this file must be stand-alone, and we don't
00190    want to have multiple files.
00191 
00192    For VALGRIND_DO_CLIENT_REQUEST, we must ensure that the default
00193    value gets put in the return slot, so that everything works when
00194    this is executed not under Valgrind.  Args are passed in a memory
00195    block, and so there's no intrinsic limit to the number that could
00196    be passed, but it's currently five.
00197    
00198    The macro args are: 
00199       _zzq_rlval    result lvalue

00205    a lot simpler.  VALGRIND_GET_NR_CONTEXT returns the value of the

00208    ppc64-linux, the R2 value at the divert point is also needed.  This

00212    guest, but guarantees that the branch instruction will not be

00216    inline asm stuff to be useful.

01152 /* This is all pretty complex.  It's so as to make stack unwinding
01153    work reliably.  See bug 243270.  The basic problem is the sub and
01154    add of 128 of %rsp in all of the following macros.  If gcc believes
01155    the CFA is in %rsp, then unwinding may fail, because what's at the
01156    CFA is not what gcc "expected" when it constructs the CFIs for the
01157    places where the macros are instantiated.
01158 
01159    But we can't just add a CFI annotation to increase the CFA offset
01160    by 128, to match the sub of 128 from %rsp, because we don't know
01161    whether gcc has chosen %rsp as the CFA at that point, or whether it
01162    has chosen some other register (eg, %rbp).  In the latter case,
01163    adding a CFI annotation to change the CFA offset is simply wrong.
01164 
01165    So the solution is to get hold of the CFA using
01166    __builtin_dwarf_cfa(), put it in a known register, and add a
01167    CFI annotation to say what the register is.  We choose %rbp for
01168    this (perhaps perversely), because:
01169 
01170    (1) %rbp is already subject to unwinding.  If a new register was
01171        chosen then the unwinder would have to unwind it in all stack
01172        traces, which is expensive, and
01173 
01174    (2) %rbp is already subject to precise exception updates in the
01175        JIT.  If a new register was chosen, we'd have to have precise
01176        exceptions for it too, which reduces performance of the
01177        generated code.
01178 
01179    However .. one extra complication.  We can't just whack the result
01180    of __builtin_dwarf_cfa() into %rbp and then add %rbp to the
01181    list of trashed registers at the end of the inline assembly
01182    fragments; gcc won't allow %rbp to appear in that list.  Hence
01183    instead we need to stash %rbp in %r15 for the duration of the asm,
01184    and say that %r15 is trashed instead.  gcc seems happy to go with
01185    that.
01186 
01187    Oh .. and this all needs to be conditionalised so that it is
01188    unchanged from before this commit, when compiled with older gccs
01189    that don't support __builtin_dwarf_cfa.  Furthermore, since
01190    this header file is freestanding, it has to be independent of
01191    config.h, and so the following conditionalisation cannot depend on
01192    configure time checks.

01195    'defined(__GNUC__) && defined(__GCC_HAVE_DWARF2_CFI_ASM)',
01196    this expression excludes Darwin.

01198    different and I haven't investigated how they work.
01199 
01200    For even more entertainment value, note we have to use the
01201    completely undocumented __builtin_dwarf_cfa(), which appears to
01202    really compute the CFA, whereas __builtin_frame_address(0) claims
01203    to but actually doesn't.  See
01204    https://bugs.kde.org/show_bug.cgi?id=243270#c47

03818 /* Several Valgrind tools (Memcheck, Massif, Helgrind, DRD) rely on knowing
03819    when heap blocks are allocated in order to give accurate results.  This
03820    happens automatically for the standard allocator functions such as
03821    malloc(), calloc(), realloc(), memalign(), new, new[], free(), delete,
03822    delete[], etc.
03823 
03824    But if your program uses a custom allocator, this doesn't automatically
03825    happen, and Valgrind will not do as well.  For example, if you allocate
03826    superblocks with mmap() and then allocates chunks of the superblocks, all
03827    Valgrind's observations will be at the mmap() level and it won't know that
03828    the chunks should be considered separate entities.  In Memcheck's case,
03829    that means you probably won't get heap block overrun detection (because
03830    there won't be redzones marked as unaddressable) and you definitely won't
03831    get any leak detection.
03832 
03833    The following client requests allow a custom allocator to be annotated so
03834    that it can be handled accurately by Valgrind.
03835 
03836    VALGRIND_MALLOCLIKE_BLOCK marks a region of memory as having been allocated
03837    by a malloc()-like function.  For Memcheck (an illustrative case), this
03838    does two things:
03839 
03840    - It records that the block has been allocated.  This means any addresses
03841      within the block mentioned in error messages will be
03842      identified as belonging to the block.  It also means that if the block
03843      isn't freed it will be detected by the leak checker.
03844 
03845    - It marks the block as being addressable and undefined (if 'is_zeroed' is
03846      not set), or addressable and defined (if 'is_zeroed' is set).  This
03847      controls how accesses to the block by the program are handled.
03848    
03849    'addr' is the start of the usable block (ie. after any
03850    redzone), 'sizeB' is its size.  'rzB' is the redzone size if the allocator
03851    can apply redzones -- these are blocks of padding at the start and end of
03852    each block.  Adding redzones is recommended as it makes it much more likely
03853    Valgrind will spot block overruns.  `is_zeroed' indicates if the memory is
03854    zeroed (or filled with another predictable value), as is the case for
03855    calloc().
03856    
03857    VALGRIND_MALLOCLIKE_BLOCK should be put immediately after the point where a
03858    heap block -- that will be used by the client program -- is allocated.
03859    It's best to put it at the outermost level of the allocator if possible;
03860    for example, if you have a function my_alloc() which calls
03861    internal_alloc(), and the client request is put inside internal_alloc(),
03862    stack traces relating to the heap block will contain entries for both
03863    my_alloc() and internal_alloc(), which is probably not what you want.
03864 
03865    For Memcheck users: if you use VALGRIND_MALLOCLIKE_BLOCK to carve out
03866    custom blocks from within a heap block, B, that has been allocated with
03867    malloc/calloc/new/etc, then block B will be *ignored* during leak-checking
03868    -- the custom blocks will take precedence.
03869 
03870    VALGRIND_FREELIKE_BLOCK is the partner to VALGRIND_MALLOCLIKE_BLOCK.  For
03871    Memcheck, it does two things:
03872 
03873    - It records that the block has been deallocated.  This assumes that the
03874      block was annotated as having been allocated via
03875      VALGRIND_MALLOCLIKE_BLOCK.  Otherwise, an error will be issued.
03876 
03877    - It marks the block as being unaddressable.
03878 
03879    VALGRIND_FREELIKE_BLOCK should be put immediately after the point where a
03880    heap block is deallocated.
03881 
03882    VALGRIND_RESIZEINPLACE_BLOCK informs a tool about reallocation. For
03883    Memcheck, it does four things:
03884 
03885    - It records that the size of a block has been changed.  This assumes that
03886      the block was annotated as having been allocated via
03887      VALGRIND_MALLOCLIKE_BLOCK.  Otherwise, an error will be issued.
03888 
03889    - If the block shrunk, it marks the freed memory as being unaddressable.
03890 
03891    - If the block grew, it marks the new area as undefined and defines a red
03892      zone past the end of the new block.
03893 
03894    - The V-bits of the overlap between the old and the new block are preserved.
03895 
03896    VALGRIND_RESIZEINPLACE_BLOCK should be put after allocation of the new block
03897    and before deallocation of the old block.
03898 
03899    In many cases, these three client requests will not be enough to get your
03900    allocator working well with Memcheck.  More specifically, if your allocator
03901    writes to freed blocks in any way then a VALGRIND_MAKE_MEM_UNDEFINED call
03902    will be necessary to mark the memory as addressable just before the zeroing
03903    occurs, otherwise you'll get a lot of invalid write errors.  For example,
03904    you'll need to do this if your allocator recycles freed blocks, but it
03905    zeroes them before handing them back out (via VALGRIND_MALLOCLIKE_BLOCK).
03906    Alternatively, if your allocator reuses freed blocks for allocator-internal
03907    data structures, VALGRIND_MAKE_MEM_UNDEFINED calls will also be necessary.
03908 
03909    Really, what's happening is a blurring of the lines between the client
03910    program and the allocator... after VALGRIND_FREELIKE_BLOCK is called, the
03911    memory should be considered unaddressable to the client program, but the
03912    allocator knows more than the rest of the client program and so may be able
03913    to safely access it.  Extra client requests are necessary for Valgrind to
03914    understand the distinction between the allocator and the rest of the
03915    program.
